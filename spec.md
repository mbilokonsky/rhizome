This is a high-level spec for the rhizome project. It contains a glossary of terms which between them seek to describe the behavior and nature of the system. It subsequently contains sections speaking in more depth to various aspects of the system.

## Glossary
* A `delta` is an immutable atomic unit that relates one or more values in semantically meaningful ways as of some point in time. A delta can be thought of as both a `CRDT` and as a `hyper-edge` in the implicit hypergraph that makes up the rhizome. A delta contains one or more `pointers`.
* A `pointer` is composed of at least two and possibly three fields. A `delta` contains a set of `pointers`. The fields of a pointer are:
	* `name` - identifies the meaning of the pointer from the perspective of the delta that contains it.
	* `target` - identifies a `value` to associate with the `name`.
	* `context` - optionally, when pointing at an `object`, the `context` identifies the field or property of that object with which this delta is associated.
* A `value` is one of two kinds of primitive that can be referred to by a `delta`:
	* a `reference` is a UUID or other value understood to be pointing at either a `delta` or an `object`.
	* a `primitive` is a literal string, number or boolean value whose meaning is not tied up in its being a reference to a larger whole.
* An `object` is a composite object whose entire existence is encoded as the set of deltas that reference it. An object is identified by a unique `reference`, and every delta that includes that `reference` is asserting a claim about some property of that object.
* A `negation` is a specific kind of delta that includes a pointer with the name `negates`, a `target` reference to another delta, and a `context` called `negated_by`.
* a `schema` represents a template by which an `object` can be compiled into a `hyperview`. A schema specifies which properties of that object are included, and it specifies schemas for the objects references by the deltas within those properties. A schema must terminate in primitive schemas to avoid an infinite regress.
	* For instance, a `hyperview` "User" of a user may include references to friends. If those friends are in turn encoded as instances of the "User" schema then all of *their* friends would be fully encoded, etc.
	* This could lead to circular references and arbitrarily deep nesting, which runs into the problem of "returning the entire graph". So our schema should specify, for instance, that the "friends" field apply the "Summary" schema to referenced users rather than the "User" schema, where the "Summary" schema simply resolves to username and photo.
* A `hyperview` is a representation of an `object` that includes a full inventory of all of the deltas that compose that object. So for instance, a hyperview of the object representing the user "Alice" might include `alice.name`, which contains an array of all deltas with a pointer whose `target` is the ID of Alice and whose context is `name`. Such deltas would likely include a second pointer with the name `name` and the target a primitive string "Alice", for instance.
	* A `hyperview` may also include nested delta/object layering. Consider `alice.friends`, which would include all deltas asserting friendship between Alice and some other person. Each such delta would reference a different friend object. In a hyperview, these references would be expanded to contain hyperviews of those friends. Schemas, as defined above, would be applied to constrain tree depth and avoid infinite regress.
* A `view` is a compression of a `hyperview` that removes delta information and flattens the structure into a standard domain object, typically in JSON. So instead of `alice.name` resolving to a list of deltas that assert the object's name it might simply resolve to `"alice"`.
	* Note that in a hyperview any property of an object necessarily resolves to a set of deltas, even if it's an empty set, because we cannot anticipate how many deltas exist that assert values on that context.
* In collapsing a hyperview into a view we may specify `resolution strategies` on each field of the schema. A resolution strategy takes as input the set of all deltas targeting that context and returns as output the value in the view. So if we have 15 deltas asserting the value for an object's name, our resolution strategy may simply say "return the target of the `name` pointer associated with the most recent delta", or it may say "return an array of names". If the value is numeric it may say "take the max" or "take the min" or "take the average".

## Deep Dives
Below are some deep dives into specific aspects of the system. These are not exhaustive, but they do provide a more detailed look at how the system is intended to work.

### On Schemas
RhizomeDB has a tripartite structure - data can exist in one of three forms.
1. **Deltas** - where each assertion made stands alone as an atomic delta that represents a relationship between arbitrary things using arbitrary semantics.
2. **Hyperviews** - where a domain node is represented as the root of a tree, and its properties are populated by deltas that reference them. The deltas then encode pointers to other objects, which in turn have properties, which map to deltas, etc.
3. **Views** - a view is a domain model, a lossy extraction of the data encoded within a hyperview but without any of the associated deltas. It's a snapshot in time and across possible interpretations.

Each of these three tiers requires a "schema" of some sort. However, it's important to differentiate between a "write-time" schema and a "query-time" schema. A delta schema determines the valid shape of a delta, it's enforced at write-time if you have it. You could also use it to filter/query the stream, but of the three types of schemas this is the only one that could be treated as a write-time constraint.

#### Schema Types
##### Delta Schemas
**Deltas** have pointers, and pointers must specify a local name under which they refer to a target on some optional context. Though the system supports arbitrary deltas, it may be desirable in many cases to constrain delta creation to a set of pre-defined functions, each of which complies with some sort of specification constraining the valid name, context or even target. We can think of this as a "Delta Schema" - for instance, you might specify a "ContainmentDeltaSchema" which says you need one pointer called "parent" whose target context is "children" and another pointer named "child" whose target context is "parent". But in a distributed system you can't actually control for canonical schemas at write time, so this has limited utility!

##### Hyperview Schemas
**Hyperviews** - this is where it gets interesting. A hyperview represents a domain model - a user, a product, an interaction, whatever - but it does so by losslessly embedding all salient deltas into a large tree. A schema here needs to be recursive - you specify your selector for your root node, and then you specify which properties on that node you want to select. But there's an additional complexity here, because you also need to specify - recursively! - the hyperview schema to apply to any domain models embedded on the far side of those deltas. If you don't do this then you run the risk of returning the whole graph. There's a further constraint here which is that the terminal nodes of your hyperview schema must terminate in primitives - think like GraphQL, you can have `User.Friends` but you don't want to recursively apply `User` to every instance of `user.friend`, you might select like ` { name, avatar }` on each friend. Same principle!

##### View Schemas
**Views** - A view schema is basically isomorphic to an application-level type. A `User` hyperview may be resolved into any number of different views - you may resolve it into a `User`, or a `Friend`, or an `Entity`, or `Alice` - whatever your application is doing. The only real constraint is that you need some way to map the properties of your view schema up to the state of your hyperview. For an early implementation goal, it's going to be a hard limit that views can only reference properties of specific hyperviews. But as we iterate on this, you can see how you *could* in theory write a view schema that side-steps the hyperview concept entirely and simply selects and reduces deltas directly. Or augments a hyperview with additional properties, or composes multiple hyperviews. All of this is feasible, but it's important conceptually to understand how these different components all come together.

#### Wave vs Particle: Partially Applied Hyperview Schemas
There's one more important fairly conceptual aspect here, and that's that because of the nature of hyperviews we are able, with rhizome, to formally reason about gaps. Your hyperview schema determines what deltas you might expect to find, but it can't enforce the existence of those deltas. So there's this notion, if you think about it, of having a `partially applied' hyperview. In a more traditional system, if you don't have values for some required property you simply don't conform to the schema and you throw a 404 or whatever.

But we can return something much more fine-grained - we can reason formally about specifically what, at the atomic delta-level, is missing from our dataset. One way to think about this is that the schema represents a wave-like value - each selected property references a general description of the *kind* of delta that may fit there, without necessarily specifying a specific delta. Once you generate the hyperview, though, you are collapsing those waves into "particles" - each delta says, "you could have had any number of different deltas, but you got ME specifically". So with a "partial application" of a hyperview schema, what you're doing is incrementally collapsing the wave form using structured and predictable boundaries - you can see where it has collapsed into "particles", and where it remains as a "wave". You can then in theory use this to identify gaps in your data, etc. This is metaphorical but it is a useful way to think about the relationship between hyperviews and their schemas.
